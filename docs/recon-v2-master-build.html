Phase 0

PRE-BUILD CLEANUP

Five codebase landmines that will corrupt the v2 build if left in place. Do this before writing a single line of new code. Nothing in v2 is safe until this is done.

Execution order matters. Items 3 + 4 are atomic — do them in one PR (they touch the same file and the null-guard depends on the array fix). Items 1 and 5 are independent. Item 2 is highest risk — it touches Firestore score persistence for all users. Do it last, after the RECON pipeline is clean and tested.
5 Items
ITEM 01 // BACKEND // 1–2 HRS Low risk
Delete src/utils/reconCompiler.js

Zero files import this anywhere in the codebase. The live compiler used by all four Barry Netlify functions is netlify/functions/utils/reconCompiler.js. The src/ version is a superseded prototype that was never deleted — and it imports db from the browser-only Firebase module. Any v2 developer who finds it and tries to import it from a Netlify function will get a runtime crash. It's not just dead — it's a trap.

ITEM 02 // BACKEND + FRONTEND // 4–6 HRS Highest risk
Consolidate 3 divergent ICP scorers into one source of truth

Three implementations, three different weight schemas, three different input field names — and ICPSettings.jsx silently overwrites every company's fit_score in Firestore each time a user saves ICP settings, replacing the server-computed score. Bonus bug: search-companies.js:737 hardcodes industry: 'Accounting' as the fallback for users with no industry ICP set.

// THREE divergent implementations found:
src/utils/icpScoring.js          // weights: industry 50%, location 25%, size 15%, revenue 10%
search-companies.js::calcFitScore // weights: industry 30pt, location 25pt, size 25pt, revenue 20pt
barry-phase2-score               // unknown weights — separate function entirely
// Fix: src/shared/icpScoring.js — pure JS, importable client + server
// normalizeCompany() resolves field aliases BEFORE any scoring runs
// scoreVersion: '2.0' written to Firestore alongside every fit_score
ITEM 03 // BACKEND // 2–3 HRS Do with Item 4
Fix the array serialization bug in compileReconForPrompt

This line appears 9 times (lines 49–129), once per section 2–10: typeof val === 'string'. Arrays return 'object', not 'string'. Every array-valued RECON field is silently discarded. Confirmed affected: coreFeatures, useCases, integrations in Section 2 alone. A user who carefully selects their use cases provides exactly the same Barry context as a user who skipped those fields.

// Before (silently drops all arrays)
if (val && typeof val === 'string') parts.push(`- ${formatKey(key)}: ${val}`);
// After (array-aware)
const coerced = Array.isArray(val)
  ? val.filter(Boolean).join(', ')
  : val;
if (coerced) parts.push(`- ${formatKey(key)}: ${coerced}`);
ITEM 04 // BACKEND // 3–4 HRS Do with Item 3
Add null/undefined guards — sanitizeReconSection()

Fixing Item 3 exposes a secondary problem: the typeof val === 'string' filter was accidentally acting as a null guard. Once replaced with array-aware handling, empty or null Firestore fields will emit blank prompt lines like - Use Cases: . A sanitizeReconSection() helper is needed that coerces arrays, trims strings, converts empty strings to null, and strips null-valued keys before Object.entries is called. Also normalize all Barry functions to || 'unknown' — zero || null patterns where the value reaches a prompt.

ITEM 05 // BACKEND // 2–3 HRS Independent
Instrument every Barry call — token logging + RECON presence flag

All four Barry Netlify functions call logApiUsage() but none log claudeResponse.usage.input_tokens or output_tokens. The section generators already do this correctly — bring the four main Barry functions in line. Add reconPresent: !!reconContext to every metadata block. Without 2 weeks of baseline data segmented by reconPresent, designing the coaching layer means guessing at both cost and personalization ROI.

Phase 1

FOUNDATION ARCHITECTURE

The data model has to be right before the experience changes. Five architectural decisions that every Phase 2 interaction depends on.

1 — reconHealth Data Structure

Store almost nothing. Compute everything at read time. Storing derived state creates race conditions — computing fresh guarantees accuracy. Only three timestamps ever get written to Firestore.

The key principle: weightedScore, gap flags, conflict flags, and freshness status are all ephemeral — computed at read time from live data. Only icpSnapshotAtLastReconSave, lastScoutSearchAt, and userRequestedReviewAt are stored. 
// Firestore: dashboards/{userId}.reconHealth — only 3 fields stored
{
  icpSnapshotAtLastReconSave: { /* icp state when Section 3 was last saved */ },
  lastScoutSearchAt: timestamp,
  userRequestedReviewAt: timestamp
}
// Computed at read time (never stored):
{
  weightedScore: 72,            // 0-100, not section count math
  criticalGaps: ['s3_industry', 's9_valueProposition'],
  staleFlags: ['s3'],            // sections not updated in >90 days
  conflictFlags: ['icp_industry_mismatch']
}
// Section weights (grounded in actual Barry usage):
// s1: 25%  s2: 20%  s3: 15%  s5: 15%  s9: 10%
// s4/s6/s7/s8/s10 split remaining 15%
Time-Based
90-day staleness

Computed at read time from completedAt timestamps. No cron jobs. No background workers.

Event-Based
ICP drift detection

Zero overlap between current ICP and the snapshot stored when Section 3 was completed triggers a conflict flag.

User-Initiated
Review timestamp

User-triggered review stored as a timestamp. Stale status computed from it at read time — not as a stored boolean.

2 — Unified ICP Scoring

Canonical location: src/shared/icpScoring.js — pure JS, no browser or Node APIs, importable from both client and Netlify functions without bundling conflicts. The scoreVersion: '2.0' field written alongside every fit_score means stale scores from old implementations can be identified and recalculated.

// normalizeCompany() resolves all field aliases BEFORE scoring runs:
function normalizeCompany(raw) {
  return {
    employeeCount: raw.estimated_num_employees || raw.company_size || raw.employee_count,
    location:      raw.company?.location || raw.state || raw.hq_location,
    name:          raw.organization_name || raw.company_name || raw.name,
    industry:      raw.industry || null   // never default to 'Accounting'
  };
}

Phase 2

INTERACTION SPEC

Three decisions that determine whether RECON ships as intended or drifts back toward a better-looking form. Every interaction in this spec is buildable and testable.

1 — Barry's Response Format

The Mirror + Inference pattern is the core interaction contract. Mirror paraphrases what Barry received — no evaluation, no score, just confirmation. Inference states what Barry will do operationally, with specificity keyed directly to confidenceImpact.

Part 1 — Mirror
Confirm what Barry received

One sentence. Paraphrases the substance — not a compliment, not a score. Demonstrates Barry read the actual input. Never starts with "Great" or "That's helpful."

Part 2 — Inference
State what Barry will do

Operational statements. +10 delta: 3–4 concrete statements with user's actual words embedded. -15 delta: 1–2 hedged statements naming what's missing. -25 delta: Gap Warning replaces this block entirely.

EXAMPLE 01 // SECTION 3 — ICP // STRONG RESPONSE // +10
Mirror
You're targeting Director-level operations leaders at Series B healthcare SaaS companies, 51–200 employees, running manual processes with legacy tooling.
Inference
I'll calibrate all Scout scoring against this profile. In Hunter, I'll lead with operational efficiency framing — not product features. Cold opens will reference healthcare compliance friction as the trigger. I'll deprioritize leads outside the 51–200 range unless their title is a strong match.
EXAMPLE 02 // SECTION 9 — MESSAGING // WEAK RESPONSE // -15
Mirror
You've described your value prop as "saving time and increasing efficiency."
Inference
I can write sequences, but I can't differentiate your messaging from generic SaaS copy without knowing the specific outcome you deliver and who experiences it. My cold opens will be functional but not sharp. Sharpen the outcome statement and I can tighten this significantly.
EXAMPLE 03 // SECTION 8 — COMPETITIVE // INCOMPLETE // -25 (GAP WARNING)
Gap Warning (replaces Inference)
Competitive context is empty. I won't reference competitor weaknesses, displacement angles, or objection-handling positioning in any Hunter output until this is filled. Prospects who ask "why you over X?" will get a generic response.
Barry never: compliments answers ("Great input!"), fabricates specifics not in the user's data, uses urgency language ("You need to fill this out"), speaks in first-person plural ("We recommend..."), or asks more than one follow-up question at a time. He's an analyst. Not a Duolingo owl.
2 — Knowledge Map State Machine

Five states. Priority when multiple conditions are true: conflict > stale > weak > empty. strong only renders when no other condition applies. The Knowledge Map moves from sidebar card to full-width front door — the primary content block below the page header.

✓
Strong
≥3 substantive fields. confidenceImpact ≥ +5. No other condition active.
—
Weak
Surface-level answers. confidenceImpact -15. Hover shows Barry's fallback assumption.
⏱
Stale
>90 days since last section update. Amber pulse animation. Timestamp shown.
!
Conflict
Scout data disagrees with RECON ICP snapshot. Highest priority state. Red indicator.
Empty
Section not started. Dashed border. No hover content until at least one field answered.
The hover state on weak sections is where trust is built. Users see Barry's exact fallback assumption — the literal string he's using in the absence of real data. Seven fallbackAssumption strings stored in src/shared/reconHealthConstants.js. Example: "Currently assuming: SMB SaaS, ACV under $10k." When that's wrong, users click immediately. Healthy urgency — no artificial pressure needed.

Barry's Current Confidence replaces the current High/Medium/Low label with a live numeric 0–100 score (the weighted reconHealth.weightedScore), color-keyed with a 400ms counter animation on change and a Δ delta showing the per-session score change.

Preview updates on threshold crossing only, via a 200ms opacity fade. No Claude API calls from the preview panel — all threshold previews are pre-authored.

The Final Test
A new user creates an account and completes RECON v2 cold. At the end of the session, they answer one question: "Does Barry actually know my business?" That is the only definition of done that matters for this entire document. Everything in Phases 0–3 was in service of that sentence.
Post-MVP: Assumption-First Dialogue. Before a user types a single word, Barry arrives pre-loaded — pulls from their domain, LinkedIn, existing CRM data — and presents working assumptions. Users confirm and sharpen, not build from nothing. They start at 60% complete. This is the experience that turns RECON from a setup step into a product differentiator. It's not in this sprint plan. It's what the next one is for.
